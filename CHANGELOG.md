# Changelog

## v0.1.0 - 2026-05-31

First tagged release. The library is correctness-verified against the
standard-library `crypto/sha256` oracle (and `golang.org/x/crypto/ripemd160` for
HASH160) and ships a real, faster SIMD backend on arm64.

### API

- Public `sha256mb` API: `Hash33` (multi-buffer fixed-33 fast path), `Lanes`,
  and `Backend`, with `Size`, `BlockSize`, and `MsgLen` constants.
- `hash160mb` subpackage: `FromPubkeys33` (batched `RIPEMD160(SHA256(x))`,
  byte-identical to `btcutil.Hash160`), `Backend`, and `Fused`.

### Backends

- Hand-tuned 4-lane arm64 hardware-SHA kernel (`sha2x4`), generated by
  `internal/shagen` and verified bit-for-bit against the scalar oracle. It
  interleaves four independent 33-byte messages through the ARMv8 SHA-256
  instructions and is **~2.6x faster than scalar** on Apple M3 (about 57.6M vs
  21.6M hashes/s at `n = 6144`, `GOMAXPROCS=1`), allocating nothing. It is the
  default on arm64.
- Portable `crypto/sha256` scalar backend, the default off arm64 and the
  correctness oracle everywhere.
- `Backend()`/`Lanes()` always report the kernel that actually runs; the library
  never advertises a SIMD backend that secretly falls back to scalar.
- amd64 SIMD (AVX-512 / SHA-NI) is not yet implemented and runs scalar; the
  `internal/shagen` generator is the reference template for adding it.

### hash160mb fused kernel

- A fused arm64 kernel (`hash160mb/internal/fusedgen`) hashes four messages with
  hardware SHA-256, hands the digests to a 4-lane NEON RIPEMD-160 entirely in
  registers (no intermediate digest buffer), and stores four HASH160 results.
- It is bit-identical to the staged path and `btcutil.Hash160` across all
  lane/tail counts and under fuzzing. On Apple M3 it ties the staged path
  single-threaded and is ~3% slower at 8 threads, so the **staged path is the
  arm64 default**; the fused kernel is selectable with `GOHASH160MB_FORCE=fused`.

### Apple-silicon feature detection

- `golang.org/x/sys/cpu` does not populate ARM64 feature flags on Darwin, which
  would silently disable the hardware kernel on every Apple-silicon Mac. SHA2
  support is assumed on `darwin/arm64` (as `crypto/sha256` itself does) and read
  from HWCAP via `x/sys/cpu` on other arm64 platforms.

### Testing

- Differential tests and fuzzing (`FuzzHash33`, `FuzzFromPubkeys33`) against the
  standard-library references; every available backend is exercised across the
  lane/tail boundary matrix and with poisoned inter-message padding that catches
  any read past byte 33.
- Contract tests: `Hash33` and `FromPubkeys33` are zero-allocation on every
  backend, safe for concurrent use, and handle message counts that are not a
  multiple of the lane width.
- Tests pass under `-race` and under forced-scalar / forced-staged / forced-fused
  execution.
